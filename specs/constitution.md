# 프로젝트 헌장 (Constitution)

> 모든 spec / plan / tasks보다 우선하는 불변 원칙.
> 이 문서를 바꾸려면 반드시 사용자의 명시적 승인이 필요하다.

## 1. 시각 기준의 단일 출처
- `docs/prototype.html`이 시각·레이아웃·인터랙션의 최종 기준이다.
- 색상·간격·모션 수치는 `docs/TOKENS.md`를 따르고, 애매하면 prototype.html의 인라인 스타일이 정답이다.
- spec 문서와 목업이 충돌하면 목업을 따르되, 충돌 사실을 사용자에게 보고한다.

## 2. Spec이 코드보다 먼저다
- spec 없는 기능은 구현하지 않는다. 구현 중 설계 변경이 필요하면 문서를 먼저 수정한다.
- spec은 "무엇을(What)", plan은 "어떻게(How)", tasks는 "순서(When)"만 다룬다. 역할을 섞지 않는다.
- 각 요구사항에는 검증 가능한 수용 기준(Acceptance Criteria)이 있어야 한다.

## 3. 데이터 경계
- 데이터 접근은 항상 repository 계층(함수)으로 격리한다. UI 컴포넌트가 데이터 출처(mock/API/IndexedDB)를 알지 못하게 한다.
- Phase 1은 mock, Phase 2에서 같은 인터페이스로 API 구현체를 교체한다.

## 4. 시간 처리
- 경과 시간은 항상 **타임스탬프 차이**로 계산한다. interval 카운트로 시간을 누적하지 않는다.
- 서버 저장 시각은 UTC(TIMESTAMPTZ), 표시 시각은 사용자 시간대(KST)로 변환한다. (Phase 2~)

## 5. 품질 게이트
- TypeScript strict 모드. `any` 사용 금지(불가피하면 사유 주석).
- 커밋 전 타입체크와 빌드가 통과해야 한다.
- 접근성: 히트영역 44px 이상, 시맨틱 태그, 키보드 포커스 가능.

## 6. 단순함 우선
- 현재 Phase에 필요 없는 추상화·범용화를 미리 만들지 않는다.
  (예: Phase 1에서 동기화 엔진, 권한 테이블, 충돌 해결 UI를 만들지 않는다 — `docs/ROADMAP.md` 참조)
- 라이브러리 추가는 필요가 증명된 뒤에만 한다.
