# 학습 관리 웹앱 — 백로그 & 개발 프로세스

> **문서 목적과 대상 독자**
> 이 문서는 이 프로젝트를 처음 접하거나 나중에 이어받을 **초급 사용자**를 위한 안내서다.

> 백로그에서 기능을 골라 Claude Code와 함께 개발을 시작하고 마무리할 수 있도록 작성했다.

## 2. 새 기능 개발 프로세스 (SDD)

```
0. 백로그   이 페이지(또는 docs/BACKLOG.md)에 아이디어 누적
            → 착수 항목 선정 + spec 번호 부여 (다음: 007)

1. Spec     specs/00N-기능명/spec.md — 요구사항 + 수용 기준(AC). "무엇을"만
            ★ 사용자 승인 후 다음 단계

2. Plan     specs/00N-기능명/plan.md — 스키마 변경, API, 컴포넌트, 상태 설계. "어떻게"
            ★ 사용자 승인

3. Tasks    specs/00N-기능명/tasks.md — 독립적으로 완료·검증 가능한 작업 단위
            (번호는 이어서: 006이 T61~T63이었으므로 007은 T71~)
            ★ 사용자 승인

4. 구현     task 1개 완료 = tasks.md 체크 + 의미 있는 단위 커밋
            커밋 전 타입체크 + 빌드 통과 확인
            spec/plan과 어긋나면 코드 우회 금지 — 문서 먼저 수정 후 진행

5. 검증     iPad(가로) 먼저 → PC 웹 → 안드로이드 폰(세로) 순서
            마이그레이션 있으면: 로컬 npm run db:setup → Neon GET /api/admin/db-setup

6. 마감     CHANGELOG.md 기록, CLAUDE.md 완료 spec 목록 갱신,
            백로그에서 항목 제거, Vercel 배포 확인
```

### 원칙 (constitution 요약)
- 각 단계 산출물은 반드시 사용자 승인 후 다음 단계로
- 데이터 접근은 repository 계층 유지 (Phase 2 API 교체 지점 명확화)
- 타이머는 startedAt 타임스탬프 기반 (setInterval 누적 금지)
- iPhone은 타깃 아님. iPad(가로) > PC 웹 > 안드로이드 폰 순으로 구현·검증

## 3. Claude Code 착수 프롬프트 모음

> Claude Code는 세션 시작 시 `CLAUDE.md`를 자동으로 읽으므로 SDD 규칙을 이미 알고 있다.
> 아래 프롬프트를 복사해서 단계별로 사용한다. `<...>` 부분만 바꾸면 된다.

### ① 백로그에서 착수 항목 선정 (선택)

```
docs/BACKLOG.md(또는 Confluence 백로그)에서 <항목명>을 다음 기능으로 착수하려고 해.
범위와 규모를 검토하고, 너무 크면 spec을 어떻게 쪼갤지 제안해줘. 아직 spec은 쓰지 마.
```

### ② Spec 작성

```
새 기능 spec을 작성하자. specs/007-<기능명>/spec.md 를 만들어줘.

기능: <한 줄 설명>
배경: <왜 필요한지>
포함: <꼭 들어갈 것>
제외: <이번에 안 할 것>

CLAUDE.md의 SDD 규칙대로 요구사항 + 수용 기준(AC)만 정의하고,
기술 설계는 넣지 마. 작성 후 내 승인을 받고 멈춰.
```

### ③ Plan 작성 (spec 승인 후)

```
specs/007-<기능명>/spec.md 승인한다. 이제 plan.md를 작성해줘.
스키마 변경(마이그레이션), API 엔드포인트, 컴포넌트 구조, 상태 관리를 포함하고
기존 repository 계층·타이머 규칙 등 CLAUDE.md 제약을 지켜. 작성 후 멈춰.
```

### ④ Tasks 작성 (plan 승인 후)

```
plan.md 승인한다. tasks.md를 작성해줘.
각 작업은 독립적으로 완료·검증 가능해야 하고, 번호는 T71부터 이어가.
작업마다 완료 조건(무엇이 되면 끝인지)을 한 줄씩 적어. 작성 후 멈춰.
```

### ⑤ 구현 (tasks 승인 후, 작업 단위 반복)

```
tasks.md 승인한다. T71부터 구현 시작해.
작업 하나 끝나면 tasks.md 체크 + 타입체크/빌드 확인 + 커밋하고 멈춰서 보고해줘.
```

다음 작업 진행:

```
T71 확인했다. T72 진행해.
```

### ⑥ 검증 & 마감 (전체 구현 후)

```
spec 007 구현이 끝났다. 마감 절차를 진행해줘:
1. 타입체크 + 빌드 최종 확인
2. 마이그레이션 있으면 로컬 db:setup 실행 안내 + Neon 반영 절차 안내
3. CHANGELOG.md에 이번 spec 변경 이력 추가
4. CLAUDE.md의 완료 spec 목록에 007 추가
5. docs/BACKLOG.md(백로그)에서 해당 항목 제거
완료 후 커밋하고, Vercel 배포 확인 방법을 알려줘.
```

### 자주 쓰는 보조 프롬프트

| 상황 | 프롬프트 |
|---|---|
| 구현 중 spec과 어긋남 발견 | `이 부분이 spec과 어긋나. 코드로 우회하지 말고 spec/plan 수정안을 먼저 보여줘.` |
| 중간에 범위 축소 | `T7x는 이번 spec에서 빼자. tasks.md와 spec.md 범위(제외)에 반영해줘.` |
| 버그 수정 (spec 불필요) | `버그: <증상>. 재현 경로: <경로>. 원인 찾아서 고치고, 커밋 전에 변경 내용 보여줘.` |
| 세션이 끊긴 후 재개 | `specs/007-<기능명>/tasks.md 보고 현재 진행 상황 파악해서, 다음 미완료 작업부터 이어서 해.` |

### 팁
- 한 세션에서 spec→plan→tasks까지 승인하며 진행하고, 구현은 작업 단위로 끊어 확인하는 흐름이 가장 안정적
- 큰 기능은 spec을 2개로 쪼개는 게 낫다 (예: "리포트" → 007 집계 API + 008 화면)
- 커밋 메시지는 기존 관례를 따른다: `feat: T71 <내용> (spec 007)`
